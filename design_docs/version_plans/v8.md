# Nullifier v8 

Delta on v7. v7 makes outputs publication-grade. v8 closes the discovery loop and lays architectural groundwork for future memory infrastructure — without committing to persistent memory at trial stage.

Two parallel goals:
1. **Closed-loop discovery** — Generator agent proposes follow-up hypotheses into a human-gated queue; approved hypotheses execute autonomously through the v7.5 pipeline.
2. **Architectural preparation** — pipeline emits data in shapes compatible with the eventual memory architecture (evidence/interpretation separation, reasoning state, in-session hypothesis families). Nothing persists across runs.

Budget limits included in v8. Batch Ensembl and dN/dS (R + PAML) shipped in v7.5.

This is the version that differentiates the tool from "yet another research copilot." It is also the version where trial-stage discipline matters most — no false data from this stage gets recycled into future investigations.

## 1. Backend changes

### Generator agent (new, Claude)
Runs after the Skeptic when v7 produces a verdict. Reads:
- The full v7 report (hypothesis verdict, analysis verdict if present, scorecard breakdown, Skeptic's alternative explanations, Compute results)
- Outlier genes flagged by the Interpreter
- Robustness findings (which genes drive verdicts)
- Unresolved conflicts from in-session reasoning state

Produces 1-5 candidate follow-up hypotheses with:
- `parent_episode_id` — pointer to the run that generated it
- `provenance` — which findings triggered the proposal
- `proposed_hypothesis` — the new claim, formalized
- `rationale` — why this follow-up matters given the parent results
- `estimated_tractability` — heuristic score, with calibration disclaimer
- `methodologist_preview` — preliminary suggestion of which tests would apply

The Generator does not execute. It writes to the queue and stops.

### Hypothesis Queue (new, in-session)
A queue of pending Generator proposals visible to the user. Each item shows:
- The proposed hypothesis
- Why it was proposed (rationale + provenance from parent episode)
- The Methodologist's preview of how it would be tested
- Approve / Reject / Edit-and-approve actions

Queue is in-session only — closed when the session ends. No cross-run persistence.

### Closed-loop execution
Approved hypotheses execute autonomously through Methodologist → Compute → Interpreter → Skeptic → Generator (potentially generating more proposals). The user is not in the loop during execution, only at proposal review.

Each iteration is a new "episode" within the same in-session hypothesis family. The Skeptic and Interpreter have access to upstream reasoning state from earlier episodes in the family.

### Hypothesis Family (in-session, soft persistence)
A family object exists within a single session:
- `family_id` — UUID, scoped to the session
- `root_hypothesis` — the original user-submitted hypothesis
- `episodes` — list of completed runs in this family, in order
- `reasoning_state` — accumulated, updated after each episode
- `status` — exploring | converging | exhausted | retired

Family dies when the session ends. The architecture is in place; persistence is not.

### Evidence / Interpretation separation (full version)
Every output object across all agents splits into two structured types:

```yaml
evidence:
  id: ev_<uuid>
  observation: { ... }   # raw retrieved or computed data
  source: ensembl|gnomad|pubmed|paml|compute_layer|...
  retrieval_timestamp: ...
  pipeline_version: v8.0.0
  used_by_interpretations: [int_<uuid>, ...]

interpretation:
  id: int_<uuid>
  draws_on: [ev_<uuid>, ...]
  claim: ...
  confidence: low | medium | high
  inference_type: direct | indirect | speculative
  produced_by: librarian | interpreter | skeptic | generator
  pipeline_version: v8.0.0
```

Every Skeptic critique, Interpreter pattern, Librarian classification, Compute result becomes either an evidence object or an interpretation that links to evidence. The Methodologist's analysis plan is itself an interpretation drawing on evidence about the data shape.

**Interpretations are re-evaluated, not blindly marked stale.** When evidence changes mid-session, the Compute layer re-runs the analyses that produced the prior interpretation and compares old result to new. Three possible statuses:

- `still_supported` — new data produces a result consistent with the prior interpretation (numerically close, same direction, same significance). The interpretation is reinforced; no user action needed.
- `contradicted` — new data produces a result that no longer supports the interpretation. UI shows warning; user can trigger reinterpretation.
- `unaffected` — interpretation drew on evidence that didn't change. No flag.

Interpretations are written as lists of atomic sub-claims so partial contradictions can be tracked per sub-claim. Display rolls them up; status tracking is granular. The deterministic Compute layer makes this work — re-running the prior test on new data and comparing numerically is what avoids asking an LLM to judge contradiction subjectively.

### Reasoning state (in-session, used within session)
Each hypothesis family carries an active reasoning state object updated after every episode:

```yaml
reasoning_state:
  family_id: <uuid>
  active_questions: [...]        # unresolved questions flagged by agents
  unresolved_conflicts: [...]    # interpretation contradictions across episodes
  current_assumptions: [...]     # what the family treats as settled (revisable)
  confidence_landscape:
    core_claim: medium
    auxiliary_claims: { ... }
  episodes_summarized: [...]
  last_updated: ...
```

The Skeptic and Interpreter read reasoning state from upstream episodes in the same family. The Generator reads it to propose meaningful follow-ups (not just outlier-driven, but tension-resolving).

The reasoning state writes itself at episode boundaries. It's structured, not free-text — fields are typed.

### Batch Ensembl fetching (performance)
Replace per-gene sequential REST calls with batch POST endpoints where available:
- `/lookup/id` accepts up to 1,000 IDs per POST → replaces 1,000 individual GET calls
- `/homology/id` accepts multiple IDs per POST → replaces per-gene ortholog fetches

Reduces ~18,000 API calls for a 3,644-gene expanded set to ~200 batched requests. First-run time on large gene sets drops from ~60 minutes to ~5-10 minutes. Cached runs unaffected (already fast).

Also add:
- `ensembl.fetching_gene N of M` progress events streamed via WebSocket so the UI shows a progress bar during large fetches
- Configurable `max_genes_for_full_analysis` (default 500) in config.toml — when the expanded set exceeds this, the Methodologist uses a stratified subsample rather than the full set. Most statistical tests don't need thousands of data points.

### Budget limits
Three-layer cost protection, all configurable in `~/.nullifier/config.toml`:
```toml
[limits]
per_run_max_usd = 0.50
per_session_max_usd = 2.00
per_day_max_usd = 10.00
per_month_max_usd = 100.00
```
- New module `tools/budget.py` — tracks rolling spend in `~/.nullifier/budget.db` (SQLite, one row per Claude call with timestamp + cost)
- Pre-call cost estimation in `llm_client.py` — estimate before sending, halt if next call would exceed any cap
- `BudgetExceededError` halts the pipeline gracefully, emits `run.halted_budget` event, preserves partial results
- Pre-run cost estimate shown in UI before user confirms
- Live remaining-budget indicator in run page footer
- Per-episode cost estimate shown at proposal-approval time in the queue panel ("Approving this will use ~$0.22, leaving $0.78 in session budget")
- Session-end summary splits cost: Claude (charged) vs Gemma (local, free)

### New events
- `generator.proposing` / `generator.proposals_complete` (with count)
- `queue.proposal_added` / `queue.proposal_approved` / `queue.proposal_rejected`
- `family.episode_complete` (with family_id and reasoning state delta)
- `interpretation.reevaluated` (with status: still_supported | contradicted | unaffected, per sub-claim)
- `ensembl.batch_progress` (N of M genes fetched)
- `budget.warning` (approaching cap) / `budget.exceeded` (cap hit)

## 2. LLM routing

| Task | Backend | Volume per session |
|---|---|---|
| Generator | Claude | 1 call per completed episode |
| Reasoning state updater | Gemma local | 1 call per episode |
| Queue item rendering (rationale generation) | Gemma local | 1 per proposal |
| Evidence/interpretation tagging | Gemma local | scales with all agents |

Generator stays on Claude — it's reasoning-heavy and the quality of proposals matters disproportionately. Reasoning state updates are structured extraction (read episode outputs, update typed fields) and route to Gemma. Same for the evidence/interpretation tagging.

## 3. Frontend changes

### Hypothesis Queue panel (new)
Sidebar on the Run page. Shows pending Generator proposals from the current session. Each card:
- Proposed hypothesis text
- Rationale (collapsible)
- Methodologist preview
- Approve / Reject / Edit buttons
- Visual link back to the parent episode that proposed it

When approved, the proposal moves to a "running" state and the queue updates with the new episode's progress streamed live.

### Family timeline (new)
A horizontal timeline showing the closed-loop progression within the current session:

```
[Root episode] → [Episode 2 (approved follow-up)] → [Episode 3] → [pending: ...]
```

Click any episode to load its report. Click a pending proposal to review it. The timeline visualizes the discovery trajectory — which is the whole point of v8.

### Reasoning state side panel
Persistent across episodes within a session. Shows current state of the active hypothesis family:
- Active questions
- Unresolved conflicts (with click-through to relevant episode pairs)
- Confidence landscape
- Current assumptions

Always visible but collapsible. This is what your father reads to understand "where the investigation is."

### Evidence/interpretation rendering
Reports gain a small toggle: "Show evidence/interpretation structure." Default off — the report renders as before. Toggle on — each claim shows its `draws_on` evidence links, and evidence objects become hoverable to see their provenance.

This is the architectural seed of the future memory system, exposed minimally in v8.

### Interpretation re-evaluation UI
When evidence changes mid-session and the Compute layer re-runs prior analyses, each affected sub-claim shows its new status:
- `still_supported` — small green badge with old-vs-new comparison on hover
- `contradicted` — yellow warning badge with "re-interpret" affordance and a numerical diff
- `unaffected` — no visual indicator

Sub-claims roll up to the parent interpretation: an interpretation with any contradicted sub-claim shows a warning at the parent level too. This makes partial contradictions visible without overwhelming the report.

### Session-end summary
When the user ends a session, a final view shows:
- All episodes run
- Generator proposals that were approved vs. rejected (and why)
- Final reasoning state of the family
- Option to export the session as JSON (no auto-persistence)

After session end, everything is gone unless the user explicitly exported.

## 5. Deferred to v9+ (memory architecture)

These are pieces from the memory architecture document that v8 deliberately does not build:

- Cross-run persistence of anything
- Theme layer
- Concept lexicon
- Counterfactual memory
- Semantic stability labels
- Multi-scale semantic compression
- Workflow performance metrics
- Agent retrieval profiles

v8 emits data in shapes compatible with these eventually being added. v9 or later activates persistence when the tool is no longer in trial stage and the data is trustworthy enough to recycle.

---

## 6. Locked decisions

1. **Generator stopping condition.** Hard cap of 5 Generator-proposed episodes per session, plus an explicit "stop generating proposals" button in the queue panel. No confidence-convergence logic. The cap applies only to autonomous loop iterations — the user can always manually submit additional hypotheses.

2. **Interpretation re-evaluation.** Compute layer re-runs prior analyses on changed evidence and produces three statuses per sub-claim: `still_supported`, `contradicted`, `unaffected`. Only `contradicted` shows a user-actionable warning. Re-interpretation is user-triggered, never automatic.

3. **Generator quality instrumentation.** Session-end summary shows count of approved vs. rejected proposals. Not persisted across sessions, but visible enough during testing that the Generator prompt can be iterated based on real signal. If approval rate is <30% in real use, the Generator prompt needs work before further investment.

## 7. Success criteria

**Closed loop works**
- Generator produces 1-5 proposals after a v7 episode completes
- Approved proposals execute autonomously through the full pipeline
- Reasoning state accumulates correctly across episodes in the same family
- User can stop the loop at any time

**Evidence/interpretation separation works**
- Every claim in every report links to specific evidence by ID
- Evidence changes mid-session trigger Compute-layer re-evaluation, producing accurate `still_supported`, `contradicted`, or `unaffected` statuses per sub-claim
- Reinforcement (still_supported) is recorded, not just contradictions
- The toggle in the UI reveals the structured graph without breaking readability

**Trial-stage discipline holds**
- Session end actually purges all in-session data
- Nothing from a session leaks into future sessions
- Export-to-JSON works for users who want to save their own work

**Performance at scale works**
- 3,644-gene expanded set completes first run in under 10 minutes (cold cache)
- Second run on same hypothesis completes in under 3 minutes (warm cache)
- Progress bar visible during Ensembl batch fetching
- Stratified subsampling activates when gene count exceeds configured max

**Budget limits work**
- Pre-run cost estimate displayed before user confirms
- Live budget indicator updates during run
- Pipeline halts gracefully when any cap is hit, with clear message and partial results preserved
- Per-episode cost estimate shown at proposal-approval time in the queue

**Generator proposal quality is at least moderate**
- On father's actual hypothesis, at least one Generator proposal is one your father would have considered worth investigating
- No proposals are obviously irrelevant or repeats of the parent hypothesis
- The Methodologist preview matches what the actual run would do
