# Nullifier v4 — Implementation Plan

A scientific hypothesis stress-testing tool, biology-first, with a multi-agent backend, hybrid Claude + local Gemma routing, real genomic data integration via Ensembl, federated literature retrieval, and a localhost web UI with live agent streaming.

This is the implementation plan. No code yet — code comes after this is locked.

---

## 1. Goals & Constraints

### Primary goal

Build a tool your father can paste a biology research proposal into and receive a structured falsification report — one that combines literature evidence and real genomic data, with visible reasoning, learnable from his corrections.

### Hard constraints

- **Biology-first.** Specialized for genomics, neuroscience, and molecular biology. Non-biology hypotheses produce a warning and run with the Analyst skipped.
- **Single user, localhost only.** No deployment, no auth, no networking concerns.
- **Hybrid model routing.** Reasoning-heavy work on Claude Sonnet 4 via API. High-volume classification on Gemma 4 E4B running locally in LM Studio.
- **Live agent streaming in the UI.** WebSocket-driven progress panel similar to Claude Code or Cursor — every tool call and agent step visible as it happens.
- **Hallucinations must be visible.** Every classification quotes the exact justifying sentence.
- **Unstudied ≠ falsified.** Novel hypotheses are reported as `NOVEL-UNTESTED`, not as weak.
- **CLI stays alongside the web UI.** Same backend modules, two surfaces.

### Non-goals (explicitly out of scope for v4)

- Multi-user support, authentication, deployment.
- Phylogenetic tree visualization (defer to v5).
- Running PAML or HyPhy directly. We use Ensembl's pre-computed dN/dS.
- Domain-general support (economics, physics). Biology-only.
- Embedding-based flag retrieval. Keyword overlap is sufficient for v4.
- Function calling / structured tool use APIs. We use plain JSON mode.

### 2.3 Configuration

A single TOML file at `~/.nullifier/config.toml`, created from `default_config.toml` on first run. Specifies:

- Claude backend (model, API key from env var)
- LM Studio backend (endpoint `http://127.0.0.1:1234/v1`, model name `google/gemma-4-e4b`, parallel request count 4, timeout 300s)
- Per-agent routing (which backend handles which task)
- Ensembl settings (rate limit, cache path, TTL)
- Literature settings (max papers per claim, parallelism)
- Flag store settings (DB path, max flags injected per run)
- Server settings (port, host = 127.0.0.1)

The user can edit this file to swap models. The `nullifier config show` CLI command prints the active config.

### 2.4 Hybrid routing

| Agent / Component | Backend | Why |
|---|---|---|
| Formalizer Stage 1 (extract core) | Claude | Reading-comprehension judgment |
| Formalizer Stage 2 (decompose) | Claude | Continuity with Stage 1 |
| Query Expander | Claude | Domain-aware variant generation |
| Librarian per-paper classification | **Gemma local** | High-volume structured task |
| Librarian synthesizer (per claim) | Claude | Cross-paper reasoning |
| Analyst (interprets Ensembl data) | Claude | Domain knowledge required |
| Skeptic | Claude | Reasoning-heavy stress test |

The Librarian per-paper step is the single biggest token sink (12 papers × N claims per run, each reading a full abstract). Routing it to local Gemma is what makes hybrid worth doing — typical run cost drops from $0.30–$0.80 to $0.10–$0.20.

### 2.5 LLM client routing layer

A single module (`llm_client.py`) exposes three functions used by all agents:

- `health_check_local()` — verifies LM Studio is reachable and the configured model is loaded. Called at server startup. Fails fast with a clear error if not.
- `llm_call_json(task_name, system, user, max_tokens)` — routes to the correct backend based on the config's `routing[task_name]`. Handles JSON parsing with one retry on parse failure.
- `llm_call_json_batch(task_name, items)` — parallel JSON calls with bounded concurrency (4 for local, 4 for Claude). Used by the Librarian per-paper step.

A token tracker is global; it accumulates Claude token usage and call counts so the report can show cost.

### 2.6 Graceful degradation

- **LM Studio unreachable at startup** → server refuses to start with a clear error message. The CLI's `nullifier serve` command exits with status 1 and tells the user how to fix it.
- **A literature source fails mid-run** → continue with the others. The report shows per-source status.
- **Ensembl down or slow** → the Analyst produces a partial report based on whatever it could fetch, with notes about which genes lacked data.
- **JSON parse failure from a model** → one retry with stronger formatting instruction; if it fails again, the call surfaces as an error in the run log and the pipeline continues.

## 3. Backend Pipeline

### 3.1 Formalizer

Two stages.

**Stage 1 (Claude):** reads the raw research-proposal text and separates falsifiable core from scaffolding. Extracts:
- `core_hypothesis` — the single empirical claim
- `cited_literature` — list of references the user already mentioned
- `proposed_methods` — what the user proposed; preserved as context, not falsified
- `starter_data` — description of any preliminary data
- `starter_entities` — list of gene/protein symbols the user provided
- `domain` — biology/neuroscience/genomics/etc.
- `key_entities` — entities at the abstract level (e.g., "synaptic genes", "BBB genes")

After Stage 1 emits, the pipeline pauses for the **confirmation gate**. In the web UI this is a modal: "Here's what I extracted as your falsifiable claim. Edit or confirm." In the CLI, it's a y/e/a prompt. The user can edit the core hypothesis before Stage 2 proceeds.

If the detected domain is not biology/neuroscience/genomics/molecular_biology/neurobiology, a warning is shown and the Analyst step will be skipped.

**Stage 2 (Claude):** decomposes the confirmed core hypothesis into atomic claims. Each claim has:
- `id` (C1, C2, …)
- `statement` — plain-English version
- `entity_a`, `entity_b` — the two subjects
- `relationship` — the verb
- `context` — scope
- `mechanism` — proposed why
- `null_hypothesis` — what observation would falsify it
- `testability` — how it could be tested with available data

Also produces `key_search_terms` for the Query Expander.

### 3.2 Query Expander

For each atomic claim, generates 5–8 diverse query variants covering:
- Direct phrasing
- Synonyms and alternative wordings
- Mechanistic terms
- Contradictory framings (to find papers that refute the claim)
- Entity-specific queries using starter gene names (e.g., `SYNGAP1 MFSD2A coevolution`)
- Adjacent concepts (for confounders)

Routed to Claude for domain-aware variant quality.

### 3.3 Federated literature retrieval

Four sources queried in parallel via thread pool:

1. **Semantic Scholar** — primary source, semantic search, citation graph, influential citation count.
2. **OpenAlex** — broad coverage, citation graph, free.
3. **Europe PMC** — full-text search across open-access papers.
4. **bioRxiv/medRxiv preprints** — accessed via Europe PMC's `SRC:PPR` filter.

Each source has a thin adapter exposing `search(query, max_results) → list[NormalizedPaper]`. Normalized fields: `source`, `id`, `doi`, `title`, `abstract`, `year`, `venue`, `citation_count`, `influential_citation_count`, `authors`.

Federated layer:
- Runs all four queries in parallel per query variant
- Dedupes by DOI first, then normalized title
- Ranks by a composite score (abstract presence + recency + citations + source preference)
- Reports per-source success/failure status for the UI

Per-claim retrieval iterates query variants until `max_papers_per_claim` (default 12) is reached.

User-cited literature is also validated: each reference the user mentioned is searched via `find_by_title` and the match (or absence) is recorded.

### 3.4 Librarian

Hybrid two-step.

**Step A (Gemma local, parallel):** Each retrieved paper is classified independently. Per-paper system prompt asks for:
- `classification` — supports / contradicts / tangential / confounder
- `justification_quote` — the exact sentence from the abstract that justifies the classification (verbatim)
- `reasoning` — brief explanation

If the model can't quote a justifying sentence, it must classify as `tangential`.

Past flags (from the flag store) relevant to the current hypothesis are injected into the system prompt as few-shot examples. Format: paper title, abstract excerpt, agent's prior classification, user's correction, reason for correction. Up to 5 flags are included.

Calls run with concurrency 4 (matches LM Studio's parallel slots).

**Step B (Claude):** Synthesizer reads all per-paper classifications for one claim and produces:
- `confounders_identified` — list of confounding factors raised by the literature
- `evidence_strength` — strong / moderate / weak / absent
- `novelty_flag` — well-studied / sparsely-studied / unstudied (critical: distinguishes "no evidence" from "evidence against")
- `literature_gap` — what isn't yet studied that would test the claim
- `synthesis` — 2–3 sentence summary of the literature state

### 3.5 Analyst

Activates only for biology domains. Produces a structured genomic evidence block.

**Step 1 — Set splitting (Claude).** The Analyst asks Claude to split `starter_entities` into two sets based on the hypothesis. For your father's case: synaptic genes (set A) and BBB genes (set B). If the hypothesis only names one set, set B is empty and cross-set analysis is skipped.

**Step 2 — Genomic data fetching (Ensembl REST, parallel).** For each gene in either set, fetch in a thread pool (4 workers, respecting Ensembl's 15 req/s limit):

| Endpoint | Purpose |
|---|---|
| `/lookup/symbol/human/{gene}` | Gene metadata: Ensembl ID, biotype, chromosome, coordinates |
| `/homology/symbol/human/{gene}?type=orthologues&target_taxon=40674` | Mammalian orthologs + pre-computed dN/dS per pair |
| `/homology/symbol/human/{gene}?type=paralogues` | Paralogs (gene family size) |
| `/genetree/member/symbol/human/{gene}` | Phylogenetic gene tree; we extract duplication events and their taxonomic levels |
| `/overlap/region/human/{region}?feature=regulatory` | Regulatory features near the gene (±5kb flank) |
| `/overlap/region/human/{region}?feature=motif` | TF binding motifs |

All Ensembl responses are cached in `~/.nullifier/ensembl_cache.db` (SQLite) for 30 days. Caching is on by default; `--no-cache` forces fresh fetches.

**Step 3 — Statistics computation (Python, no LLM).** For each set:
- Mean ortholog count across mammals
- Mean and stdev of dN/dS values, pooled across all orthologs of all genes in the set
- Mean paralog count
- Mean duplication count from gene trees
- Outlier detection: genes with dN/dS > 2 stdev from the set mean

Cross-set:
- Jaccard index of TF motif overlap between the two sets
- Set of shared TFs
- Set of TFs unique to each set

**Step 4 — Interpretation (Claude).** Sends the raw data and computed statistics to Claude with a prompt that:
- Asks for observed patterns relative to the hypothesis
- Asks for outlier genes worth flagging
- Asks for an interpretation of the regulatory overlap
- **Requires explicit limitations**: dN/dS is pairwise human-vs-X (not branch-specific), regulatory overlap is Jaccard not statistically normalized, the Analyst is observational not a rigorous phylogenetic comparative method
- Returns `overall_genomic_assessment`: supports / neutral / contradicts / inconclusive

### 3.6 Skeptic

The reasoning-heavy stress test. Sees:
- The core hypothesis and atomic claims
- Per-claim Librarian assessments (synthesizer output)
- The top 3 raw abstracts per claim (for sanity-checking the Librarian)
- The Analyst's full output (set stats, cross-set analysis, interpretation)

Required to consider biology-specific alternative explanations (the prompt enumerates them):
- **Allometric scaling** — apparent correlation explained by both things scaling with a third variable like brain or body size
- **Shared developmental program** — co-expression because of shared upstream developmental TFs, not co-evolution
- **Pleiotropy** — single gene affecting both phenotypes
- **Phylogenetic non-independence** — correlated traits in related species look like co-evolution but require PGLS to test properly
- **Selection bias in cited literature**
- **Genetic drift / neutral evolution** — correlation arising by chance

Scores each from 1 (falsified) to 10 (strongly supported):
- statistical_robustness
- literature_consensus
- mechanistic_plausibility
- counter_explanation_risk (inverted: 10 = survives alternatives well)
- genomic_evidence_alignment (new for v4: how well the Ensembl data backs the claim)
- novelty_adjusted_confidence (penalize confidence when novelty_flag is "unstudied")
- overall_falsifiability_score

Verdict: `STRONG | MODERATE | WEAK | FALSIFIED | NOVEL-UNTESTED`. Use `NOVEL-UNTESTED` when most claims are unstudied — the hypothesis isn't supported or refuted by existing literature; it needs primary investigation.

Also produces:
- Top 3 alternative explanations with plausibility ratings and how to rule each out
- A decisive experiment: specific method + dataset + expected outcome under H1 vs. H0
- A librarian sanity-check note: did the Librarian's classifications seem correct based on the raw abstracts the Skeptic saw?

## 4. Streaming Architecture

### 4.1 Event types

The pipeline emits events to a queue. The FastAPI WebSocket endpoint forwards them to the connected browser. Event categories:

- **Lifecycle:** `run.started`, `run.complete`, `run.error`
- **Stage:** `stage.formalizer.start`, `stage.formalizer.complete`, etc. for each agent
- **Confirmation gate:** `formalizer.awaiting_confirmation` (pauses, waits for `confirm` or `edit` message from client)
- **Tool calls:** `tool.literature_search` (with query, source), `tool.literature_search.complete` (with paper count), `tool.ensembl_request` (with endpoint, gene), `tool.ensembl_request.complete`
- **Granular progress:** `librarian.paper_classified` (with paper N of M, paper title, classification result), `analyst.gene_fetched` (with gene N of M)
- **Token usage:** periodic `tokens.update` events with running totals

Each event has: `type`, `timestamp`, `run_id`, `payload`.

### 4.2 WebSocket protocol

The browser connects to `ws://127.0.0.1:8000/ws/runs/{run_id}` after starting a run. The server pushes events as JSON. The browser sends control messages back: `confirm_hypothesis`, `edit_hypothesis`, `cancel_run`.

### 4.3 UI rendering

The `ProgressPanel` component shows a vertical timeline. Each agent gets a section that expands as events arrive:

```
✓ Formalizer        ─ 2.3s, 1,847 tokens
  └─ Extracted core hypothesis
  └─ Awaiting confirmation [confirmed by user]
  └─ Decomposed into 3 atomic claims

✓ Query Expander    ─ 1.1s, 894 tokens
  └─ C1 → 7 query variants
  └─ C2 → 6 query variants
  └─ C3 → 8 query variants

⟳ Librarian         ─ running…
  ├─ Searching literature (federated)
  │  ├─ ✓ Semantic Scholar (15 papers)
  │  ├─ ✓ OpenAlex (12 papers)
  │  ├─ ✓ Europe PMC (8 papers)
  │  └─ ✓ bioRxiv (3 papers)
  ├─ Per-paper classification (Gemma local)
  │  └─ 7 of 12 classified…
```

Tool calls render as compact cards similar to Claude Code: collapsed by default showing the tool name and a one-line summary, expandable to show full input/output. The card style: muted background, monospace tool name, click to expand.

Token totals update live in a footer strip showing running cost.

### 4.4 Cancellation

The cancel button sends `cancel_run` over the WebSocket. The pipeline checks a cancellation flag at every agent boundary and at every paper/gene iteration. On cancellation, the run is marked `cancelled` in the runs DB and partial results are preserved.

## 5. Data & Persistence

### 5.1 SQLite databases

Three local DBs, all in `~/.nullifier/`:

| DB | Purpose |
|---|---|
| `runs.db` | Run history (input hypothesis, report JSON, timestamps, status) |
| `flags.db` | User-flagged classification corrections |
| `ensembl_cache.db` | Cached Ensembl responses, 30-day TTL |

### 5.2 Runs DB schema

A `runs` table:
- `run_id` (UUID, primary key)
- `created_at`
- `completed_at`
- `status` — running / completed / cancelled / error
- `input_text` — original hypothesis text
- `formalized_json` — Stage 1 + Stage 2 output
- `report_json` — full pipeline output (formalized + evidence + analyst + verdict)
- `token_usage_json` — per-backend token counts
- `verdict` — denormalized for quick filtering in the history view
- `core_hypothesis` — denormalized for the history list display

A separate `events` table for replaying past runs in the timeline view:
- `run_id`, `seq`, `timestamp`, `event_type`, `payload_json`

### 5.3 Flag store schema

(Unchanged from v3.) A `flags` table:
- `id`, `created_at`
- `hypothesis_summary`, `domain`, `entities_json`
- `paper_title`, `paper_abstract_excerpt`
- `agent_classification`, `agent_justification`
- `user_classification`, `user_reason`

### 5.4 Flag retrieval (across-session learning)

When a new run starts, the Librarian queries the flag store for relevant past flags by:
1. Filtering by exact domain match
2. Scoring remaining flags by entity overlap with the current run's entities
3. Adding a fallback score from hypothesis-keyword overlap
4. Taking the top 5

These are formatted as few-shot examples and injected into the per-paper classifier's system prompt. The UI shows a "X past corrections applied to this run" indicator so the user sees the learning happening.

## 6. Frontend

### 6.1 Stack

- **Vite + React + TypeScript**
- **Tailwind CSS** for styling
- **shadcn/ui** for primitives (Button, Card, Dialog, Toast, Tabs, etc.)
- **Recharts** for the scorecard bars and Analyst charts
- **lucide-react** for icons
- No router library needed — three views, conditional rendering on a top-level state is enough. (Or use `react-router-dom` if it feels cleaner; either works.)

### 6.2 Pages

#### Home page

A clean single-purpose surface:
- Large textarea for hypothesis input (auto-grows, accepts pasted formatted text)
- Dropdown of recent inputs (last 5 from runs DB) for re-running
- "Run analysis" button, disabled until input is non-trivial
- Sidebar: link to History, link to Flags, link to Config

#### Run page

Active during a run and after completion. Two-pane layout:

**Left pane — Live progress timeline.** Vertical stream of agent stages and tool calls. Auto-scrolls to the latest event. Each tool call is a collapsible card. Token + cost counter at the bottom.

**Right pane — Report (renders progressively as data arrives).** Sections:

1. **Confirmation modal** appears overlaying the right pane after Formalizer Stage 1 completes. User edits if needed, then confirms.
2. **Hypothesis card** — shows the confirmed core, domain, entity counts.
3. **Atomic claims table** — each claim with its null hypothesis.
4. **Evidence summary table** — per claim: supports / contradicts / tangential / confounders counts, evidence strength, novelty flag.
5. **Genomic evidence panel** (Analyst output):
   - Bar chart: dN/dS distributions per set (set A, set B, control if available). Recharts box-plot or histogram.
   - Sortable ortholog table: gene × species, with dN/dS, % identity. Filterable.
   - TF motif overlap: a simple list with a Jaccard index badge. Shared TFs called out.
   - Outlier genes panel with one-line explanations.
   - Limitations callout box in muted yellow.
6. **Alternative explanations** — three cards, color-coded by plausibility (high = red, medium = yellow, low = green).
7. **Stress test scorecard** — one row per scoring category with a horizontal bar (Recharts BarChart, single-value).
8. **Verdict card** — large, color-coded by verdict (STRONG green, MODERATE yellow, WEAK orange, FALSIFIED red, NOVEL-UNTESTED blue).
9. **Decisive experiment** — highlighted panel.
10. **Skeptic's sanity-check** — small muted panel noting whether Librarian classifications looked right.
11. Bottom action bar: "Review classifications" button (jumps to flag review), "Export JSON" button, "View raw events" button (opens a dev panel showing the full event log).

#### History page

Three-column list:
- Date / time
- Truncated hypothesis (first 80 chars)
- Verdict badge

Click → opens a read-only Run page with the report rendered from cached JSON. Replay button optionally re-streams the events from the events table to recreate the live experience.

Filtering by verdict and by date.

#### Flag review page

Two layouts depending on whether you're reviewing a specific run or browsing all flags.

**Per-run review** (entered from the Run page's "Review classifications" button):
- One paper at a time, full-width card
- Shows: paper title, abstract excerpt, year, venue, agent's classification, justification quote, reasoning
- Action row: keep / flag with correction. If flagged, dropdown of correct classifications + optional reason text field.
- Keyboard shortcuts: `j` next, `k` previous, `f` flag, `enter` confirm correction.
- Progress indicator: "12 of 47 reviewed, 3 flagged."

**Flag library** (browsed from the home sidebar):
- List of all past flags
- Filter by domain, by classification correction (e.g., "show only cases where I corrected 'supports' → 'contradicts'")
- Each flag shows the agent's mistake and the user's correction with reason
- Read-only; flags are permanent records of training data.

### 6.3 Component design notes

- **Tool call cards** — use shadcn `Card` with a small monospace header (`literature_search`, `ensembl.get_orthologs`), a one-line summary visible by default, expandable to show input/output JSON. Status icon: spinner while running, checkmark when complete, X on error. Style mirrors Claude Code/Cursor — quiet, scannable, never loud.
- **Confirmation modal** — uses shadcn `Dialog`. Shows extracted hypothesis in an editable textarea, "Confirm" and "Cancel" buttons. Ratifies the most important moment in the pipeline.
- **Charts** — Recharts only. No fancy library wrappers. Bar charts and tables. Keep visual density high but not crowded.
- **Loading states** — every section that hasn't received data yet shows a skeleton matching its eventual shape. No spinners-on-empty-pages.
- **Error states** — every agent's section can render an error card if the agent failed. The pipeline continues past failed steps where possible.

### 6.4 What the UI does not have

Worth being explicit about scope discipline:

- No phylogenetic tree visualization (defer)
- No interactive scatter plots beyond what Recharts gives by default
- No Venn diagrams (use a list with Jaccard index instead)
- No multi-user features, no sharing, no auth
- No editing of past flags (they're a permanent record)
- No editing of past runs (immutable history)
- No dark/light mode toggle for v4 (default to whatever shadcn ships with)

## 7. Backend API

### 7.1 REST endpoints

| Method | Path | Purpose |
|---|---|---|
| `POST` | `/api/runs` | Submit a hypothesis, returns `run_id` |
| `GET` | `/api/runs/{run_id}` | Fetch full report (after completion) |
| `GET` | `/api/runs` | List past runs (paginated) |
| `DELETE` | `/api/runs/{run_id}` | Delete a run |
| `GET` | `/api/runs/{run_id}/events` | Fetch all events for replay |
| `POST` | `/api/runs/{run_id}/flag` | Submit a flag from review mode |
| `GET` | `/api/flags` | List all flags, with filters |
| `GET` | `/api/health` | Health check (verifies LM Studio reachable, API keys set) |
| `GET` | `/api/config` | Read active config (sanitized — no API keys) |

### 7.2 WebSocket endpoint

`WS /ws/runs/{run_id}` — connect after creating a run. The server immediately sends any buffered events, then streams new events live. Client can send `confirm_hypothesis`, `edit_hypothesis`, `cancel_run` messages back.

### 7.3 Server lifecycle

`nullifier serve` (CLI subcommand) starts the FastAPI server on `127.0.0.1:8000` and the Vite dev server on `127.0.0.1:5173`. Health check on LM Studio runs at startup; if it fails, the server prints a clear error and exits.

For local distribution simplicity, the Vite frontend builds to static assets that FastAPI serves directly. So in practice your father runs one command (`nullifier serve`), opens `http://127.0.0.1:8000`, and gets the UI. The dev-server-with-HMR setup is only used during development.

## 10. Open Questions to Resolve Before Coding

These should be locked in before implementation starts.

1. **Should the confirmation gate also let the user edit `starter_entities`?** The Analyst depends heavily on this list being correct. If Stage 1 misses a gene name or misclassifies one, the Analyst either skips it or splits the sets wrong. Letting the user edit the list at confirmation time is a small UX add but a significant correctness add. Recommend yes.

2. **What's the control set for the Analyst's dN/dS comparison?** Right now the Analyst computes set A vs. set B stats. To answer "are these genes evolving differently than the baseline?" you need a third set: brain-expressed genes that aren't synaptic or BBB-related. Three options:
   - Hardcode a small control set (e.g., 20 housekeeping genes) shipped with the tool
   - Pull a control set dynamically from Ensembl via GO terms
   - Skip the control comparison for v4 and add it in v5
   Recommend hardcode a small control set for v4 — about 20 genes. It's minimal effort and meaningfully improves the dN/dS interpretation.

3. **How aggressive should the Librarian's per-paper truncation be?** Gemma E4B has 128K context but the per-paper call is one paper per prompt, so context isn't the bottleneck. Truncating abstracts to ~2000 chars feels right. Confirm.

4. **Should past runs include the actual retrieved papers, or just the metadata?** Storing all retrieved abstracts inflates the runs DB. Storing only metadata means re-running won't reproduce exactly. Recommend: store metadata only in the DB; if the user wants the abstracts, they can re-run with cache enabled (literature search isn't cached, but that's a v5 problem).

5. **Should the UI's history view replay past runs as a live stream, or just dump the report immediately?** Both have merit. Recommend: report renders immediately, with a "Replay timeline" button that streams the events at original-or-faster speed for those who want to see how the agents reasoned.

6. **CORS and security for the localhost server.** Since it's localhost only and single-user, recommend permissive defaults: bind to `127.0.0.1` only, no auth, CORS allows the Vite dev origin. Document in README that the server should never be exposed publicly.

7. **What happens when a user re-runs the same hypothesis?** Should we detect duplicates and offer to show the past run instead, or just always run fresh? Recommend: always run fresh, but the Home page's "recent inputs" dropdown surfaces past runs so the user can choose to view rather than re-run.

## 11. Testing Strategy

Lightweight for a prototype but worth being explicit:

- **Unit tests** on the literature dedup logic, the Ensembl response parsers, the flag retrieval scoring.
- **Integration tests** on a tiny example: a 3-line hypothesis, mocked literature responses, mocked Ensembl responses, verify the full pipeline produces a valid report.
- **Smoke test** at startup: every time `nullifier serve` runs, after health checks pass, run a single mocked end-to-end pipeline call with all real model backends to catch routing/auth/parsing breakage.
- **Manual regression**: keep the synapse-BBB example and verify it runs cleanly after every significant change.

No frontend tests for v4 — single user, internal tool, low value. If it ships and is used, add Playwright tests in v5.

## 12. What Success Looks Like

This prototype succeeds if your father can:

1. Paste his hypothesis into the web UI without formatting issues.
2. See the extracted core hypothesis in the confirmation modal and either accept it or fix it.
3. Watch the pipeline run live, with each agent's progress visible, in 1–3 minutes.
4. Receive a falsification report that contains:
   - At least 2 atomic claims correctly decomposed
   - 8–12 retrieved papers per claim with sentence-quoted classifications
   - A genomic evidence section with real Ensembl data on his starter genes
   - Sharp, biology-specific alternative explanations from the Skeptic (allometric scaling and phylogenetic non-independence should appear by name)
   - A verdict that correctly distinguishes "novel and untested" from "weak evidence"
5. Click into flag review, correct 5–10 misclassifications in a few minutes.
6. Re-run the same hypothesis and see the "X past corrections applied" indicator change his results in some visible way.
7. Find the second run's output noticeably better than the first's.

If 1–6 work and 7 doesn't, the architecture needs reconsidering — but at least we'll know exactly where it broke. If 1–7 all work, we have something worth turning into a product.
