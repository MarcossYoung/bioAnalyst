# Nullifier v9 — Epistemic Robustness

Delta on v8. v8 closed the discovery loop and introduced the evidence/interpretation schema. v9 uses that schema to make the system's biological reasoning *auditable* and *grounded*.

One coherent story: every claim Nullifier produces should be checkable against the source it cites, and every domain-classification step should pull its few-shot exemplars from a single inspectable library rather than from prose buried in agent prompts.

Two pillars:
1. **Cross-source contradiction detection** — a deterministic check that interpretations are entailed by the evidence they claim to draw on, catching synthesis drift before it reaches the user.
2. **Biomedical exemplar library** — Layer-2 structured task objects (SRTP sense) that consolidate scattered few-shot examples into a single, evaluable, biology-expert-editable resource.

Both depend on v8 being in place. v9 is where the falsification dimension of the system becomes a measurable property rather than a stylistic hope.

---

## 1. Why this version exists

The Nullifier architecture has structural guards against the worst failure modes of LLM-based literature analysis — the verbatim quote rule, the deterministic Compute layer, the Skeptic's independent re-check. v8 added the evidence/interpretation schema, which makes every claim point at the data it came from.

What none of those mechanisms catch is **synthesis drift**: the Librarian classifies a paper as "supports the claim" with a verbatim quote, but the *summary sentence* the Librarian writes about that paper subtly misrepresents what the quote says. The Skeptic reads the synthesis, not the underlying corpus, and propagates the drift. The user reads the Skeptic. By the time anyone notices, the report has been treated as evidence in a downstream decision.

This is the failure mode the audience cares most about — gene biologists and PIs will not forgive confident misrepresentation of source literature. v9 builds the check.

## 2. Backend changes

### Contradiction detector (new, Gemma local)
A deterministic-shaped classification step that runs after the Librarian and after the Skeptic. For every interpretation produced in the pipeline, it:

1. Looks up the interpretation's `draws_on` evidence list (from v8 schema)
2. For each evidence object, extracts the verbatim quote(s) or numerical result that grounds it
3. Runs an NLI-style classification on each (interpretation claim, source quote) pair:
   - `entails` — the source supports the claim
   - `contradicts` — the source actually undermines the claim
   - `neutral` — the source is unrelated or insufficiently specific
4. Emits a `contradiction_report` artifact, structured per sub-claim (the atomic sub-claim decomposition from v8 makes this granular)

The detector does *not* produce judgment about whether a contradiction is substantive — it produces the signal. The Skeptic reads the report as additional structured input and decides whether the flagged contradictions are real (genuine source misrepresentation) or artifacts (wording differences that don't change meaning).

### Why this routes to local
NLI is bounded, classification-shaped, runs in volume (every interpretation × every evidence link), and the per-call cost dominates. It fits the existing "high-volume cheap classification, low-volume expensive reasoning" routing principle. Routing to Claude would cost more than the rest of the pipeline combined.

### Why it does not run on general NLI models
Off-the-shelf NLI models (trained on SNLI/MNLI) handle everyday-English entailment well but fail systematically on biomedical text. The entailment "PSD-95 was elevated" → "a postsynaptic density protein was elevated" requires knowing that PSD-95 *is* a postsynaptic density protein. The entailment "reduced dN/dS at synaptic loci" → "purifying selection acts on synaptic genes" requires knowing that low dN/dS *means* purifying selection. These relationships are absent from general training data, leading to both false negatives (missed contradictions — the worse failure mode) and false positives (spurious flags the Skeptic must dismiss).

v9 starts with **few-shot Gemma with curated biomedical exemplars**, drawing from the exemplar library described below. If accuracy plateaus, the next move is **ontology-grounded NLI**: retrieve relevant GO terms, SynGO categories, and HGNC synonyms before classification, using infrastructure already present in `gene_sets.py` and `ensembl.py`. A separate biomedical NLI model (BioNLI, SciNLI) is the third fallback but adds operational surface area and is not the first option.

### Contradiction report schema

```yaml
contradiction_report:
  id: cr_<uuid>
  pipeline_version: v9.0.0
  produced_at: ...
  interpretation_id: int_<uuid>
  episode_id: <uuid>
  pairs_checked:
    - sub_claim: "..."           # one atomic sub-claim from the interpretation
      evidence_id: ev_<uuid>
      source_quote: "..."        # verbatim quote or numerical result
      label: entails | contradicts | neutral
      detector_confidence: low | medium | high
      detector_rationale: "..."  # short Gemma-generated explanation
  summary:
    total_pairs: N
    entails: N
    contradicts: N
    neutral: N
    flagged: [<list of pair indices where label == contradicts>]
```

The Skeptic reads this artifact in addition to the inputs it already takes in v8. The Skeptic's verdict logic gains an explicit step: any interpretation with a `contradicts` flag is examined and either upheld (Skeptic judges the flag a wording artifact) or downgraded (Skeptic judges the contradiction substantive). Skeptic decisions are recorded in the verdict output.

### Eval set for the detector
Built in parallel with the detector, not after. 50-100 hand-labeled pairs drawn from actual Nullifier runs (v7 and v8 archived runs are the source material). Each pair carries:
- `sub_claim` text
- `source_quote` text
- `human_label` (entails/contradicts/neutral)
- `domain_tags` (synaptic / BBB / cross-species / regulatory / methodological)
- `difficulty` (trivial / requires-domain-knowledge / ambiguous)

The eval set lives at `backend/nullifier/eval/contradiction_nli/` and is version-controlled. Detector accuracy is reported per domain tag and per difficulty bucket — a detector that's 90% accurate on trivial pairs and 40% on requires-domain-knowledge pairs is worse than one that's 75% across the board, and a single accuracy number would hide that.

### Skeptic verdict gains a new dimension
The Skeptic's seven-dimensional scoring is augmented with an eighth: **source fidelity**. Defined as the fraction of interpretations in the report whose sub-claims are all `entails` against their cited evidence (after Skeptic adjudication of detector flags). A run with high comprehensiveness but low source fidelity is no longer a passing run — the verdict is downgraded with a `SOURCE-FIDELITY-LOW` flag, analogous to v7's `RESULTS-PROBLEMATIC`.

### New events
- `contradiction_detector.checking` (with count of pairs)
- `contradiction_detector.complete` (with summary stats)
- `skeptic.source_fidelity_assessed` (with score and flag count)
- `verdict.fidelity_downgrade` (when a verdict is downgraded due to low source fidelity)

## 3. Biomedical exemplar library

### Motivation
Few-shot examples are scattered across individual agent prompts in v7 and v8 — the per-paper classifier has its own, the gene-set relevance scorer has its own, v8 adds three more surfaces (Generator, reasoning-state updater, evidence/interpretation tagger), and v9 adds the contradiction detector. Without consolidation, the same gene name appears as an exemplar in five different prompts with five slightly different framings; updates require hunting through agent files; and the exemplars are not evaluable as a set because they don't exist as a set.

The flag store (`flag_store.py`) already implements exemplar-injection for one task (per-paper classification, fed by user corrections). The exemplar library generalizes that pattern to all classification tasks in the pipeline.

### Structure
A single TOML or Python module at `backend/nullifier/exemplars/biomedical/`, organized by task:

```
exemplars/biomedical/
├── __init__.py
├── README.md                    # how to contribute exemplars
├── per_paper_classification.toml
├── gene_set_relevance.toml
├── contradiction_nli.toml       # new in v9
├── entity_extraction.toml       # for future background-literature ingestion
├── evidence_interpretation_tagging.toml   # added by v8 integration
└── shared/
    ├── synonyms.toml            # PSD-95 ↔ DLG4, ApoE ↔ APOE, etc.
    ├── hierarchies.toml         # synaptic protein → postsynaptic protein → PSD-95
    └── domain_failure_modes.toml  # common confounders by domain
```

Each task file is a list of input-output pairs with metadata:

```toml
[[per_paper_classification]]
input.abstract = "..."
input.claim = "..."
output.label = "supports"
output.quote = "..."
output.rationale = "..."
domain_tags = ["synaptic", "cross-species"]
added_by = "father:2026-04-12"   # provenance: human, date
notes = "edge case: paper supports the molecular mechanism but contradicts the cross-species generalization"
```

### Prompts pull from the library at construction time
Agent prompts in v9 no longer embed exemplars inline. They construct their prompt by:
1. Identifying the task (per-paper classification, contradiction NLI, etc.)
2. Pulling the exemplar set for that task from the library
3. Optionally filtering by domain tag if the current hypothesis carries domain hints from the Formalizer
4. Injecting the filtered exemplars into the prompt template

The flag store remains the *user-correction* exemplar source. The library is the *curated* exemplar source. Both feed into the same prompt-construction pipeline; the library provides the baseline behavior and the flag store provides correction over time.

### Why this is a Layer-2 object in the SRTP sense
The SRTP framing distinguishes reasoning policy (Layer 1), structured task objects (Layer 2), reasoning graphs (Layer 3), and runtime (Layer 4). Exemplars are structured task objects: they define what the agent is supposed to *recognize* in its domain, declaratively, in data, separate from the orchestration and execution. Pulling them out of prompts and into a library makes the domain knowledge inspectable — a new contributor or auditor can read `gene_set_relevance.toml` and see *exactly* what the system has been taught about gene-set relevance, without reading code.

This is the kind of consolidation the SRTP document explicitly asks for: converting hidden domain knowledge from procedural code into declarative semantic objects.

### Library curation workflow
Three sources of exemplars:
1. **Hand-curated by domain experts** — direct edits to TOML files, typically when a known failure mode is identified
2. **Lifted from user flags** — periodic export of high-quality flag-store entries into the curated library (one-way, with attribution)
3. **Lifted from contradiction-detector eval set** — when a labeled NLI pair turns out to be highly informative (the detector consistently fails on it before being trained), it becomes an exemplar

All three sources carry provenance metadata so origin is auditable.

### Library is evaluable
For every task that pulls from the library, v9 adds a small eval harness: swap exemplar set A for set B, run the task on a held-out evaluation set, measure accuracy. This makes the exemplar library a first-class engineering artifact with metrics attached, not a folder of prose.

## 4. LLM routing

| Task | Backend | Volume per run |
|---|---|---|
| Contradiction detector (per-pair NLI) | Gemma local | ~20-80 calls per episode |
| Skeptic source-fidelity adjudication | Claude | folded into existing Skeptic call |
| Exemplar library lookup | n/a (data fetch) | per agent call |
| Eval harness runs | Gemma local | offline, batched |

Contradiction detection is the volume hot path and stays local. Adjudication of flagged contradictions stays with the Skeptic and adds no new Claude calls — the Skeptic already runs once per episode and the contradiction report becomes one more input to its existing prompt.

## 5. Frontend changes

### Source fidelity badge on every interpretation
Every interpretation in the report gains a small fidelity indicator:
- Green check — all sub-claim/evidence pairs labeled `entails`
- Yellow warning — at least one pair labeled `contradicts` but Skeptic judged it non-substantive (with hover showing the pair and the Skeptic's reasoning)
- Red warning — at least one pair labeled `contradicts` and Skeptic judged it substantive (with hover showing the pair, the Skeptic's reasoning, and the resulting verdict downgrade)

This makes synthesis drift visible at the granularity it occurs.

### Contradiction inspector
A new view, accessible from any interpretation with a non-green fidelity badge, that shows:
- The sub-claim text
- The cited source quote (verbatim)
- The detector label and rationale
- The Skeptic's adjudication
- An action: "this looks wrong" (which writes a flag-store entry that can be lifted into the exemplar library later)

### Exemplar library viewer (admin/contributor-facing)
A read-only browser at `/exemplars` that lists tasks, exemplar counts per task, domain-tag distribution, and provenance. Contributors edit the TOML files directly; the UI surfaces what's there.

### Source-fidelity column in the verdict scorecard
The seven-dimensional scorecard from v7 gains an eighth column. A run with `SOURCE-FIDELITY-LOW` shows the dimension prominently and links into the contradiction inspector.

### Run history filters
The History page gains a filter: "show only runs with source-fidelity issues." This is the primary way the developer-user (you) finds runs worth using to expand the eval set and the exemplar library.

- Sample 60-80 sub-claim/source-quote pairs from archived v7/v8 runs

- This day exists before the detector itself — without the eval set, detector quality is unmeasurable

**Day 3 — Contradiction detector (Gemma)**
- Few-shot prompt construction pulling from `exemplars/biomedical/contradiction_nli.toml`
- Initial exemplar set: 10-15 hand-crafted pairs covering synonym, hierarchy, method-result equivalence, and false-positive cases
- Detector wired in after Librarian and after Skeptic, emits `contradiction_report`
- Baseline accuracy measured against the eval set, reported per domain tag and difficulty

**Day 4 — Skeptic integration**
- Skeptic prompt updated to read `contradiction_report` as structured input
- Source-fidelity dimension added to scorecard
- `SOURCE-FIDELITY-LOW` verdict flag wired into existing verdict logic
- Verdict downgrade behavior tested on synthetic runs (inject known contradictions, verify downgrade fires)

**Day 5 — Frontend**
- Source-fidelity badges on interpretations
- Contradiction inspector view
- Updated verdict scorecard
- Run history filter for fidelity issues
- Exemplar library read-only viewer

**Day 6 — Iteration on detector quality**
- Run detector against the full eval set, identify failure clusters by domain tag
- Add targeted exemplars to `contradiction_nli.toml` for failing clusters
- Re-measure accuracy; iterate until the failure modes that matter most (false negatives in requires-domain-knowledge bucket) are below a defined threshold

**Day 7 — Father test**
- End-to-end on a real hypothesis with known contradictions seeded in the source corpus
- Verify the detector catches them, the Skeptic adjudicates correctly, the UI surfaces them legibly
- Verify exemplar library additions during testing flow back into the library cleanly

**Day 8 (buffer)** — If detector accuracy on the requires-domain-knowledge bucket is unacceptable, this day is for moving to ontology-grounded prompts (retrieving GO/SynGO/HGNC facts before classification). The infrastructure for this is already present from `gene_sets.py` and `ensembl.py` — the work is prompt engineering and one additional retrieval step in the detector pipeline.

## 7. Locked decisions

1. **Detector failure-mode priority.** False negatives (missed contradictions) are weighted more heavily than false positives (spurious flags) in detector tuning. Missed contradictions are silent failures the Skeptic cannot recover from; spurious flags are noise the Skeptic actively dismisses. Prompt iteration and exemplar additions prioritize reducing false negatives in the requires-domain-knowledge bucket.

2. **Skeptic owns adjudication.** The contradiction detector produces signal; the Skeptic produces judgment. Detector flags do not automatically downgrade verdicts. Only Skeptic-confirmed substantive contradictions trigger `SOURCE-FIDELITY-LOW`. This preserves the existing "deterministic signal, reasoning agent judges" architecture.

3. **Exemplar library is the single source.** After v9 ships, no agent prompt embeds a few-shot exemplar inline. All exemplars come from the library or the flag store, both via the prompt-construction pipeline. New exemplars added via direct TOML edits, flag-store lifts, or eval-set lifts — all carry provenance.

4. **Library evaluation is per-task, per-domain-tag.** A single accuracy number across all tasks is rejected as a metric. Each task reports accuracy broken down by domain tag and difficulty. The exemplar library is judged on whether it improves the *worst* bucket, not the average.

---

## 8. Success criteria

**Contradiction detection works**
- Detector accuracy on the eval set is at least 80% on the `requires-domain-knowledge` difficulty bucket and at least 90% on the `trivial` bucket
- False-negative rate (missed contradictions) is below 10% across all difficulty buckets
- Detector runs add less than 15% to total per-episode wall-clock time
- Cost per episode (detector calls only) is under $0.02 in expected real usage

**Source fidelity is visible**
- Every interpretation in every report has a fidelity badge
- `SOURCE-FIDELITY-LOW` verdicts trigger correctly on synthetic seeded contradictions
- The contradiction inspector lets a user trace any flagged claim to its source quote in under three clicks

**Exemplar library functions as a first-class artifact**
- All scattered exemplars from v7/v8 are migrated into the library with no behavioral regression
- The library has at least one exemplar set per active classification task in the pipeline
- A non-developer (your father, or another domain expert) can add a new exemplar by editing a TOML file without touching any agent code
- The eval harness produces measurable accuracy improvements when exemplar additions target known failure clusters

**Falsification dimension is no longer a stylistic hope**
- A Nullifier run that synthesizes incorrectly from its sources produces a low source-fidelity score
- A Nullifier run that synthesizes correctly from its sources produces a high source-fidelity score
- The distinction is reproducible across reruns of the same input

- Persistent memory architecture — the v8 architectural-preparation work activates here, not before

v9 ships the falsification dimension. v10 builds memory on top of it.
