# Nullifier v7.1 — Bugfix Plan

Bugfix release on v7. No new features. The goal is for v7 to actually meet its stated success criteria before v7.5 (PAML) or v8 (closed-loop) build on top.

The v7 spec required every test result to have effect size, CI, and correction method. The recent test run shows that's not happening. v7.1 makes the v7 contract actually hold.

---

## 1. Bugs to fix

### A. Compute layer — Mann-Whitney post-hoc returns empty

When Kruskal-Wallis is significant, the pipeline runs pairwise Mann-Whitney with correction. Currently returns no results (test output shows blank rows). Likely causes:
- Pairwise comparison loop receives empty group lists or wrong shape
- Correction method applied to empty results
- Result object built but not returned, or returned without populated fields

Fix:
- Trace the call path from Methodologist plan → Compute layer execution of Mann-Whitney post-hoc
- Confirm groups passed in are non-empty and correctly shaped
- Confirm the result object populates all required `TestResult` fields
- Add a regression test: significant Kruskal-Wallis must produce non-empty post-hoc results

### B. Compute layer — Spearman correlation errors silently

Test output shows "spearman: error" with no further detail. Likely causes:
- Null values in input arrays
- Mismatched array lengths
- Insufficient data points (Spearman needs n ≥ 3)
- Exception caught and stringified as "error" instead of returning a structured error

Fix:
- Locate the Spearman wrapper in `tools/compute.py`
- Add explicit pre-checks: drop nulls, verify matched lengths, verify minimum n
- If pre-checks fail, return a structured `TestResult` with explicit `skipped` status and reason — not an opaque "error" string
- Add a regression test for each failure mode (null inputs, length mismatch, insufficient n)

### C. Compute layer — Confidence intervals missing on all tests

The v7 spec required CIs as a non-optional field. Currently every test result shows "—" in the CI column. Either CIs aren't being computed, aren't being returned, or aren't being rendered.

Fix:
- Audit every test wrapper in `tools/compute.py`. Each must compute a 95% CI or return an explicit `{value: null, reason: "..."}` for tests where CI isn't applicable (Fisher's exact, for example)
- Where bootstrap CIs are needed (Cliff's delta, Cohen's d, epsilon-squared), use scipy.stats bootstrap or write a simple resampling loop with 1000 iterations
- Schema validation at Compute layer boundary should reject any result missing the CI field
- Add a regression test: every test result schema must validate before being returned

### D. Frontend — "correction: [object Object]" rendering bug

The correction method object is being passed to the frontend without being stringified or destructured properly. JavaScript is rendering the object literal.

Fix:
- Find where correction method is rendered in `ComputeResultsSection.tsx`
- Either render the structured object as JSON-formatted text, or extract the relevant fields (method name, threshold) and render them explicitly
- Add a smoke test: render the compute results section with a real test result and confirm no "[object Object]" strings appear

### E. Frontend — Outlier genes show name plus period, no explanation

Output shows "SPOCK1 — ." instead of the explanation text. Either the Interpreter isn't producing the explanation field, or the frontend isn't reading it correctly.

Fix:
- Trace whether the Interpreter's outlier gene output includes the `why_notable` and `implication` fields
- If the Interpreter is producing them, fix the frontend rendering in `GenomicPanel.tsx`
- If the Interpreter isn't producing them, fix the Interpreter prompt to require these fields
- Either way, add validation that outlier gene records have non-empty explanations before they reach the renderer

### F. dN/dS — display the truth about Ensembl coverage

Currently dN/dS appears as "n/a" for both gene sets without explanation. Users see this and may assume it's a transient data gap.

Fix (small, no PAML yet — that's v7.5):
- Wherever dN/dS would render in the report or compute layer output, add a note: "Ensembl pairwise dN/dS no longer populated in recent Compara releases. Branch-specific dN/dS via PAML is planned for v7.5."
- Update the Interpreter prompt to acknowledge this when discussing evolutionary rate
- Update the Skeptic prompt to not penalize the report for missing dN/dS — it's a known data limitation, not a methodological gap

### G. Verdict robustness — gracefully handle absent primary tests

Currently shows "Not applicable — no primary tests" when tests are broken. Strictly true but misleading — robustness is unavailable because tests failed, not because no tests were defined.

Fix:
- Distinguish three states in the robustness panel:
  - "Robustness check ran successfully" (verdict stable/sensitive/fragile)
  - "Robustness check skipped — primary tests had insufficient results to perturb"
  - "Robustness check not applicable — no primary statistical tests in this run"
- Update the renderer to show the correct message based on actual state

## 2. v7 success criteria — actually validated this time

Before v7.1 is declared done, run the synapse-BBB hypothesis through the full pipeline and confirm:

- [ ] Every test result has effect size AND 95% CI (or explicit `null` with reason for tests where CI is N/A)
- [ ] Significant Kruskal-Wallis produces populated Mann-Whitney post-hoc results
- [ ] Spearman either returns a result or returns structured `skipped` with reason — never opaque error
- [ ] Frontend renders all results without "[object Object]" anywhere
- [ ] Outlier genes show real explanations, not periods
- [ ] dN/dS unavailability is explained in the report rather than appearing as silent "n/a"
- [ ] Robustness panel shows the correct status message for the run's actual state

This list becomes the v7.1 acceptance test. No release without all boxes checked.

## 4. Not in v7.1

Explicitly out of scope:
- PAML integration (that's v7.5)
- OrthoMaM static dataset (also v7.5 — making the dN/dS limitation a fixed problem rather than a documented one)
- Closed-loop discovery (v8)
- Memory architecture preparation (v8)
- Budget limits (v8)
- New tests beyond what v7 already specified

If a bug seems to need new functionality to fix, that's a signal it belongs in v7.5 or v8, not v7.1.

---

## 5. Process lesson for future versions

v7 was declared done before its own success criteria were validated. v7.1 fixes the symptoms; the process gap that allowed it is the deeper issue.

Going forward: every version's "Success criteria" section becomes a literal checklist run against a test hypothesis before the version is closed. v7.1 itself is the first version where this discipline applies. v7.5 and v8 inherit it.
