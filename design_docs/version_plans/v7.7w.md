# Nullifier v7.7 -- W1 + W2 (guardrails)

## Context

The synapse/BBB run gave a genuinely unstudied hypothesis a confident, wrong-shaped genomic
verdict, and reported "12 paper(s) retrieved, none classified" for every claim. Two
independent defects caused this, both cheap to fix, and fixing them immediately stops the
tool from emitting confident-but-wrong output:

- **W1 -- Librarian classification reliability:** papers are retrieved but the per-paper
  classifier fails on every paper, and that failure reads as a literature void
  (`NOVEL-UNTESTED`) instead of a tool failure.
- **W2 -- Methodologist construct-validity gate:** a test family is selected without checking
  it matches the claim's structure; a mismatched test returns a clean p-value that gets
  laundered into a confident `contradicts`.

W1 and W2 are independent and ship together. W2 converts the false `contradicts` into an
honest `untestable`; the real test behind that gate is built later in **v7.7w3.md** (W3).

> **Naming discipline (keep verbally separate everywhere):** W1 is per-paper *classification*
> (label a paper supports/contradicts/tangential) and *citation matching* (does a database hit
> actually correspond to the cited paper). W2 is *construct/test selection* (does the chosen
> statistic match the claim's structure). Never conflate "classification" and "construct" in
> code or comments.

## W1 -- Librarian classification reliability

### Confirmed root cause (validated against code AND the run_190002 export)
The classifier did **not** classify successfully and then get judged accurate -- it failed
outright on every paper, and the failure was misread as a confirmed literature void.

- In `run_190002_export.txt`, every claim has `classifications: []` and every paper sits in
  `failed_classifications` with the same error:
  `Error code: 400 - {'error': "'response_format.type' must be 'json_schema' or 'text'"}`.
  This is **not** "LM Studio down" and **not** a JSON parse failure -- it is an incompatible
  `response_format` request. The local classification call is therefore broken **100% of the
  time** whenever it is routed to local, independent of whether LM Studio is running.
- `tools/llm_client.py` `llm_call_json_batch(...)`: each per-paper call is wrapped in
  `try/except Exception -> return {"_error": str(e)}`. No re-raise, no surfacing. The 400 from
  the local server becomes a silent `_error`.
- Routing: `config/default_config.toml [routing] librarian_per_paper = "local"`.
- `agents/librarian.py`: results with `"_error"` (or non-dict) are pushed into an
  **already-existing `failed_classifications` list** (`{paper_id, paper_title, error}`) and
  `continue`d. `classifications` ends empty; the synthesizer receives the fallback
  `("(no papers classified)",)`. The per-claim evidence dict already carries `classifications`,
  `failed_classifications`, and `retrieved_papers`.
- The Skeptic's `librarian_sanity_check` then declared the empty classifications a *confirmed
  absence* of relevant literature (verdict region of the export), turning a total classifier
  failure into "accurately unstudied." That inference is invalid.
- `frontend/.../EvidencePanel.tsx` renders "N paper(s) retrieved, none classified" when
  `classifications` is empty but `retrieved_papers` is non-empty -- identical rendering whether
  we found nothing relevant or could not classify what we found.

So most of the data needed already exists (`failed_classifications`); it is simply never
fixed at the source, never categorized, and never surfaced.

### Fixes
1. **Fix the local `response_format` request (the actual bug)** (`tools/llm_client.py`
   `_call_local_json`). The request currently sends a `response_format.type` the local server
   rejects (must be `json_schema` or `text`). Send a supported value -- either a proper
   `{"type": "json_schema", "json_schema": {...}}` matching the per-paper classification schema,
   or `{"type": "text"}` plus the existing JSON-fence stripping. This alone restores non-zero
   classifications when the local model is up, and is the primary correctness fix.
2. **Categorize drop reasons** (`agents/librarian.py` + a small helper). Map each
   `failed_classifications[*].error` string to a `drop_reason` enum:
   `api_schema_error` / `response_format_error` (HTTP 400 / invalid `response_format` -- the
   error actually seen here), `model_unreachable` (connection refused / timeout),
   `parse_error` (JSON decode), `empty_response`,
   `quote_mismatch` (only if verbatim-quote enforcement actually discards -- **verify**: current
   code defaults a missing `classification` to `"tangential"`, so confirm whether quote
   enforcement is a real drop path before adding this bucket), `other`. Compute per-claim counts
   `{retrieved, classified, dropped, drop_reasons:{...}}`.
3. **`classifier_degraded` event + flag** (`events.py`). Emit per-claim when
   `dropped / retrieved` exceeds a threshold. Thread a top-level
   `evidence["classifier_degraded"]` boolean (+ per-claim counts) so downstream consumers see
   it. (Optional safety net for genuine local outages: reroute that claim's classification to
   Claude; secondary to fix #1, not the centerpiece.)
4. **Distinguish void from failure in the report** (`report/renderer.py`, `EvidencePanel.tsx`,
   `lib/types.ts`). Show per-claim `retrieved / classified / dropped` with reasons.
   "0 classified because the classifier failed" must render visibly differently from
   "papers classified, none relevant."
5. **Skeptic must not read failure as void** (`agents/skeptic.py`). When
   `evidence["classifier_degraded"]` is set, the verdict path cannot be `NOVEL-UNTESTED` on the
   strength of an empty literature axis, and `librarian_sanity_check` must not declare a
   literature void; it notes the degradation, and the literature axis is scored low-confidence
   rather than "absent -> novel."

### W1b -- Citation match quality (separate guardrail, same Librarian surface)
A distinct bug surfaced in the same run: user-cited biology papers were "validated" against
completely unrelated database hits. Example from the export: the cited *"Functionalization of a
Protosynaptic Gene Expression Network -- In the Light of Evolution"* was recorded in
`cited_literature_validated` as a match to *"Radiation Resistant Camera System for Monitoring
Deuterium Plasma Discharges in the Large Helical Device"* (DOI `10.1585/pfr.15.2402039`, a
plasma-physics paper) -- accepted with no similarity check.

- **Fix:** in the cited-literature validation step (`agents/librarian.py` / `tools/literature.py`
  `find_by_title`), gate a `database_match` on a minimum similarity between the user reference
  (title/description) and the candidate hit (title/abstract) -- e.g. normalized token overlap or
  an embedding/similarity threshold. Below threshold, mark the citation `unverified` (do not
  record it as `validated`) and surface it as such. A wrong-domain match must never be presented
  as a validated citation.

### Touchpoints
`tools/llm_client.py`, `agents/librarian.py`, `tools/literature.py`,
`config/default_config.toml`, `events.py`, `report/renderer.py`,
`frontend/src/components/EvidencePanel.tsx`, `frontend/src/lib/types.ts`, `agents/skeptic.py`.
(`tools/flag_store.py` unchanged -- but verify the flags few-shot injection is not itself
producing parse failures.)

### Done when
- With the local model up, a run yields **non-zero classifications** (the `response_format`
  fix works); the per-paper 400 error no longer appears.
- A run with the local model genuinely unavailable produces a visible `classifier_degraded`
  path -- **not** a silent `NOVEL-UNTESTED`, and the sanity check does not call it a void.
- The report shows per-claim retrieved/classified/dropped counts with drop reasons (including
  `api_schema_error`).
- A cited paper that only matches an unrelated database hit is marked `unverified`, not
  `validated`.

## W2 -- Methodologist construct-validity gate

### Confirmed current behavior (validated against code)
- Atomic claim schema (`agents/formalizer.py`, stage2): `{id, statement, null_hypothesis, ...}`.
  **No construct field.**
- `agents/methodologist.py` `run_methodologist(...)`: LLM picks a plan, then `primary_tests` is
  filtered to entries whose `test` is in `TEST_LIBRARY`. No check that the test's question
  matches the claim's structure.
- `tools/compute.py` `TEST_LIBRARY` entries are `{fn, kind}`; `_test_result(...)` already
  supports `available=False`, `skipped=True`, `skip_reason=...` -- so an `untestable` axis has a
  native representation.
- `agents/skeptic.py` `_format_analyst_for_skeptic(...)` already has a neutral path
  ("Not available - score genomic_evidence_alignment as 5 (neutral)") when no analyst result --
  the hook for an `untestable` genomic axis to score neutral instead of contributing to
  falsification.

### Fixes
1. **Tag each claim with a construct** (`agents/formalizer.py`). Add a `construct` OutputField
   to `FORMALIZER_STAGE2_SPEC` and validate it in normalization. Allowed values:
   `set_difference` (do groups differ on a property -- current tests valid here),
   `cross_lineage_rate_correlation` (do two gene sets co-vary in rate across the tree -- needs
   W3's mirrortree_lite), `phenotype_association` (per-species phenotype tracks a genomic
   quantity -- needs PGLS, **tag and defer**). Default to `set_difference` when the LLM omits it.
2. **Construct->method capability map + gate** (`agents/methodologist.py` + per-test
   `constructs` metadata in `compute.TEST_LIBRARY`). Tag each test with the construct(s) it can
   answer: `kruskal_wallis`/`mann_whitney_posthoc`/`spearman`/`pearson` -> `set_difference`;
   `paml_branch_model` -> lineage-specific selection (not the cross-set correlation construct);
   `mirrortree_lite` -> `cross_lineage_rate_correlation` (registered in W3). The methodologist
   may only keep tests whose `constructs` include the claim's tag. If none remain, return
   `{"untestable": true, "untestable_reason": "...", "required_construct": "..."}` with empty
   `primary_tests`.
3. **Propagate `untestable` through compute and skeptic.** Compute runs nothing for an
   untestable axis (or emits `_test_result(available=False, skipped=True, skip_reason=...)`).
   The Skeptic's genomic axis returns `untestable` -- never `contradicts`/`supports` -- and
   `genomic_evidence_alignment` scores neutral (5). A non-significant result from a
   construct-mismatched test must not contribute to falsification.
4. **Saturation gate (cheap, lands here)** (`agents/analyst.py` `_set_statistics`, consumed by
   `interpreter.py`/`skeptic.py`). Compute `dnds_saturation_fraction` = fraction of usable
   dN/dS values within ~|v - 1.0| < 0.01 (alignment saturation, not biology). When it exceeds a
   threshold (~0.5), flag the genomic axis low-confidence/`untestable` regardless of test choice
   -- the run's "no significant difference" is mostly absence of measurement and must not read
   as absence of effect.

### Touchpoints
`agents/formalizer.py` (construct tag), `agents/methodologist.py` (capability map + gate),
`tools/compute.py` (per-test `constructs`; honor untestable), `agents/compute.py`
(`run_compute` skip), `agents/analyst.py` (saturation fraction), `agents/skeptic.py` (genomic
axis can return `untestable`, neutral score), `agents/interpreter.py` (surface gate reason),
frontend verdict rendering + `lib/types.ts`.

### Done when
- The synapse/BBB run returns genomic axis = `untestable`
  (construct = `cross_lineage_rate_correlation`, no matching method until W3) instead of
  `contradicts`.
- A true `set_difference` claim still routes to and runs Kruskal-Wallis/Spearman (no regression).
- A high-saturation gene set triggers the low-confidence/`untestable` flag.
