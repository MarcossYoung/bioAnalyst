# Nullifier v5 — Implementation Plan

A delta on v4. v4 is built and working. v5 adds two things:

1. **Unified hypothesis + completed analysis input.** The tool now stress-tests claims at any stage — pre-analysis, mid-analysis, or post-analysis. A single pipeline adapts based on what the user provides.
2. **First-class flag UI.** Flag review and the flag library, properly wired into the web UI rather than half-implemented or CLI-only.

Everything else from v4 stays. Same backend modules, same hybrid routing, same Ensembl integration, same WebSocket streaming, same biology-first focus.

---

## 1. Why v5 exists

After running v4 on a real hypothesis, two things became visible:

**One — the tool's value depends on when in the research workflow it's used.** Used on a bare hypothesis (your father's first pass), v4 correctly returns `NOVEL-UNTESTED` and stops there. Useful but limited. The user then goes and does the analysis themselves, generating real findings the tool never sees. The natural next step — feed those findings back in for stress-testing — has no path in v4. You either re-paste the original hypothesis (and get the same `NOVEL-UNTESTED` verdict) or you have no tool at all.

**Two — the flag system got built as backend but not as UX.** The flag store works. Flags inject into the Librarian. But your father can't actually flag anything from the UI in v4, which means the learning loop never actually engages in practice.

v5 closes both gaps.

---

## 2. What changes in v5 — the short version

**Backend changes:**
- Formalizer Stage 1 extraction schema gains two optional fields: `completed_analysis` and `methods_used`.
- Confirmation gate event payload gains a per-section structure (hypothesis / methods / results / starter_data) instead of just hypothesis.
- Skeptic system prompt gains a conditional block activated when `completed_analysis` is present, requiring methods/statistics/interpretation evaluation.
- Skeptic output schema gains four optional scorecard rows that only appear when completed analysis exists.
- Analyst gains a "reported-vs-retrieved" comparison sub-step: when the user reports specific statistics (dN/dS means, sample sizes, p-values), the Analyst attempts to verify them against Ensembl-retrievable data.

**Frontend changes:**
- Confirmation modal redesigned to show each extracted section with keep/edit/remove controls.
- Report renderer conditionally shows the methods/statistics critique sections.
- Flag review page wired end-to-end into the Run page and accessible from history.
- Flag library page implemented with filtering.
- "Flags applied: N" indicator visible during runs.

**No changes:**
- Pipeline architecture (Formalizer → Query Expander → Literature → Librarian → Analyst → Skeptic)
- Hybrid Claude/Gemma routing
- WebSocket event protocol (events stay the same shape, only payload contents differ slightly)
- SQLite databases and schemas
- The CLI

## 3. Backend changes — detailed

### 3.1 Formalizer

The Stage 1 system prompt extends to extract two new optional fields. The extraction schema becomes:

```
{
  "core_hypothesis": "...",
  "cited_literature": [...],
  "proposed_methods": [...],          // what user proposed but hasn't done
  "methods_used": [...],              // NEW: methods user actually ran (if results are reported)
  "starter_data": "...",
  "starter_entities": [...],
  "completed_analysis": {             // NEW: optional; null if input has no results
    "findings": [
      {
        "label": "...",               // e.g., "Finding 1 — Synaptic genes more constrained"
        "claim": "...",               // the specific empirical claim made
        "statistics_reported": [
          {"test": "Kruskal-Wallis", "value": "p=1.95e-12", "context": "dN/dS across sets"},
          ...
        ],
        "sample_sizes": {"set_a": 68, "set_b": 58, "control": 36},
        "interpretation": "..."        // how user interpreted the result
      }
    ],
    "datasets_used": ["SynGO", "gnomAD", "Ensembl", "..."],
    "limitations_acknowledged": ["..."]  // what the user explicitly flagged
  },
  "domain": "...",
  "key_entities": [...]
}
```

**Detection logic:** the prompt instructs the model to set `completed_analysis: null` when the input contains only a hypothesis/proposal. It populates the field when the input contains explicit results (sentences like "we found that...", "the analysis showed...", numerical statistics, named tests like Kruskal-Wallis, table data).

**The Formalizer must distinguish three states explicitly:**
- *Pure hypothesis* — `completed_analysis = null`. Tool runs as v4 did.
- *Hypothesis + proposed methods* — `proposed_methods` populated, `methods_used = []`, `completed_analysis = null`. Tool runs as v4 did. (This was your father's first input.)
- *Hypothesis + completed analysis* — `methods_used` populated, `completed_analysis` populated. Tool activates v5 critique behaviors.

The detection isn't perfect. The confirmation gate is the safety net.

### 3.2 Confirmation gate — restructured

In v4, the confirmation gate showed only the core hypothesis with keep/edit/abort. In v5, the gate becomes section-based.

The WebSocket event `formalizer.awaiting_confirmation` payload changes from:

```
{ "core_hypothesis": "..." }
```

to:

```
{
  "sections": [
    {"id": "hypothesis", "label": "Core hypothesis", "content": "...", "removable": false},
    {"id": "methods_proposed", "label": "Proposed methods", "content": [...], "removable": true},
    {"id": "methods_used", "label": "Methods used", "content": [...], "removable": true},
    {"id": "starter_entities", "label": "Starter genes", "content": [...], "removable": true},
    {"id": "completed_analysis", "label": "Completed analysis findings", "content": {...}, "removable": true}
  ],
  "detected_state": "pure_hypothesis | hypothesis_with_proposed_methods | hypothesis_with_results"
}
```

Each section has `keep`, `edit`, or `remove` controls. The hypothesis section is not removable (a run requires a hypothesis). Everything else is optionally removable, which is your father's escape hatch when the Formalizer misclassifies something.

The client response to the gate becomes:

```
{
  "action": "confirm",
  "edited_sections": {
    "hypothesis": "...",         // if user edited
    "completed_analysis": null   // if user removed
  }
}
```

The pipeline then proceeds with the confirmed-and-edited extraction.

### 3.3 Skeptic — conditional critique behavior

The Skeptic's system prompt is split into two parts:

**Always-active block (v4 behavior):**
- Identify top 3 alternative explanations (allometric scaling, phylogenetic non-independence, pleiotropy, etc.)
- Score the original v4 categories
- Produce verdict and decisive experiment

**Conditional critique block (new in v5, activated only when `completed_analysis` is non-null):**
- Evaluate each reported finding's statistical rigor
- Specifically check for: sample size adequacy, multiple testing correction, phylogenetic independence (PGLS for comparative dN/dS), effect size vs. p-value distinction, test appropriateness for the data type
- Cross-reference reported statistics against retrievable data (where the Analyst has done the reproducibility check)
- Identify interpretation overreaches — places where the conclusion exceeds what the data supports

The Skeptic's output schema gains four optional fields (`null` when no completed analysis):

```
"methods_critique": {
  "appropriate_tests": [...],
  "questionable_tests": [
    {"test": "...", "concern": "...", "severity": "minor|moderate|major"}
  ],
  "missing_controls": [...]
},
"statistical_critique": {
  "sample_size_concerns": [...],
  "multiple_testing_status": "corrected|uncorrected|partially_corrected",
  "effect_sizes_reported": "yes|partial|no",
  "phylogenetic_independence_addressed": "yes|no|not_applicable"
},
"reproducibility_check": {
  "verified_findings": [...],
  "unverifiable_findings": [...],
  "discrepancies": [{"finding": "...", "reported": "...", "retrieved": "..."}]
},
"interpretation_critique": {
  "supported_conclusions": [...],
  "overreach_concerns": [{"conclusion": "...", "concern": "..."}]
}
```

The scorecard gains four optional rows that render only when completed analysis is present:
- `methods_appropriateness` (1–10)
- `statistical_rigor` (1–10)
- `reproducibility_of_reported_findings` (1–10)
- `interpretation_validity` (1–10)

These contribute to `overall_falsifiability_score` when present.

The verdict set gains one option: `RESULTS-PROBLEMATIC` — used when the hypothesis itself might be supportable but the user's specific analysis has methodological issues serious enough that the results shouldn't be trusted as evidence either way. This is distinct from `WEAK` (literature contradicts) or `NOVEL-UNTESTED` (no literature exists).

### 3.4 Analyst — reproducibility check sub-step

When `completed_analysis` is present and reports specific statistics on genes in the starter set or on standard databases, the Analyst attempts a reproducibility check.

**What it can verify:**
- Reported dN/dS values for individual genes (compare against Ensembl pairwise dN/dS)
- Reported ortholog counts
- Reported paralog counts
- Reported regulatory feature counts

**What it can't verify:**
- Branch-specific dN/dS (Ensembl only has pairwise)
- LOEUF scores (would require gnomAD integration — see v6 candidate features below)
- Custom statistical tests (Kruskal-Wallis on user's gene sets)
- Phylogenetic timing (520 Ma / 450 Ma waves)

The Analyst's output adds a `reproducibility_check` block when applicable:

```
"reproducibility_check": {
  "verifiable_claims": [
    {"claim": "synaptic mean dN/dS = 0.027", "verifiable": "partially", "note": "..."}
  ],
  "discrepancies": [...],
  "limitations": [
    "Branch-specific dN/dS cannot be verified against Ensembl pairwise data",
    "LOEUF scores require gnomAD integration not yet implemented"
  ]
}
```

This feeds into the Skeptic's `reproducibility_check` synthesis.

### 3.5 Pipeline state machine

The pipeline gains a small branching point after the Formalizer and confirmation gate:

```
Formalizer Stage 1
  ↓
Confirmation Gate (section-based)
  ↓
Has completed_analysis? ──── no ──→ v4 path (unchanged)
  │
  yes
  ↓
v5 critique path:
  ↓
Formalizer Stage 2 (atomic claims — still runs)
  ↓
Query Expander
  ↓
Federated Literature Retrieval
  ↓
Librarian (per-paper local + Claude synthesizer)
  ↓
Analyst (with reproducibility check sub-step)
  ↓
Skeptic (with critique block active)
  ↓
Report (with critique sections rendered)
```

The only structural difference is the reproducibility check inside the Analyst and the conditional Skeptic block. Everything else is the same pipeline.

### 3.6 New event types

Three new WebSocket events for the streaming UI:

- `formalizer.detected_completed_analysis` — fires when Stage 1 finds results in the input, payload includes a brief description of what was detected
- `analyst.reproducibility_check.start` and `analyst.reproducibility_check.complete` — bracket the new sub-step
- `skeptic.critique_mode_active` — fires when the Skeptic activates the conditional critique block, signals to the UI that critique-specific scorecard rows will be populated

No changes to existing event types.

## 4. Frontend changes — detailed

### 4.1 Confirmation modal — restructured

The current v4 modal shows the core hypothesis in an editable textarea with Confirm/Cancel. The v5 modal becomes a multi-section interface.

Layout:

```
┌─ Confirm extracted sections ─────────────────────────┐
│                                                       │
│  Detected: Hypothesis + Completed analysis            │
│  [Edit detected state ▼]                              │
│                                                       │
│  ┌─ Core hypothesis ────────────────── [edit] ─────┐ │
│  │  We hypothesize that synaptic genes...          │ │
│  └─────────────────────────────────────────────────┘ │
│                                                       │
│  ┌─ Methods used (12) ───────── [edit] [remove] ──┐ │
│  │  • Ensembl REST for ortholog retrieval         │ │
│  │  • gnomAD GraphQL for LOEUF scores             │ │
│  │  • PAML codeml for branch-specific dN/dS       │ │
│  │  • ...                                          │ │
│  └─────────────────────────────────────────────────┘ │
│                                                       │
│  ┌─ Starter genes (8) ─────────── [edit] [remove] ┐ │
│  │  Set A: SYP, SYNGAP1, SYNGR1, CAMK2A, DLG4     │ │
│  │  Set B: MFSD2A, SLC2A1, SPOCK1                  │ │
│  └─────────────────────────────────────────────────┘ │
│                                                       │
│  ┌─ Completed analysis (6 findings) ─ [edit] [remove]┐│
│  │  Finding 1 — Synaptic genes more constrained    │ │
│  │    Kruskal-Wallis: p=1.95e-12 on dN/dS          │ │
│  │  Finding 2 — Primate-specific BBB acceleration  │ │
│  │  ...                                             │ │
│  └─────────────────────────────────────────────────┘ │
│                                                       │
│  [Run analysis]  [Cancel]                             │
└──────────────────────────────────────────────────────┘
```

**Behaviors:**

- Each section card has its own state: `kept`, `edited`, `removed`.
- Clicking `edit` expands the card into an editable view (textarea for prose, list editor for arrays, structured form for completed_analysis findings).
- Clicking `remove` collapses the section to a "Removed — [restore]" stub.
- The hypothesis card has no `remove` button (the pipeline requires a hypothesis).
- The "Detected" badge at the top is editable — if the user wants to override the auto-detected state (e.g., "this isn't really a completed analysis, treat it as a hypothesis"), the dropdown forces the state.
- The whole modal is keyboard-navigable. Tab cycles between cards. `Enter` on a section's edit button opens edit mode. `Esc` collapses edit mode. `Cmd/Ctrl+Enter` confirms and runs.

The confirmation modal becomes the single most important interaction surface in the tool. It's where misclassifications get caught before the downstream agents waste tokens on them.

### 4.2 Report — conditional critique sections

When the pipeline runs in v5 critique mode, the report grows new sections rendered after the existing v4 sections:

**Methods critique panel** — renders if `methods_critique` is non-null. Shows:
- Appropriate tests with green checkmarks
- Questionable tests with severity badges (minor / moderate / major)
- Missing controls callout

**Statistical rigor panel** — renders if `statistical_critique` is non-null. Shows:
- Sample size concerns per finding
- Multiple testing status badge (corrected / uncorrected / partially_corrected)
- Effect size reporting indicator
- Phylogenetic independence indicator (when relevant — only for comparative analyses)

**Reproducibility check panel** — renders if `reproducibility_check` is non-null. Shows:
- A simple table: finding | reported value | retrievable value | match status
- Discrepancies highlighted in yellow or red

**Interpretation critique panel** — renders if `interpretation_critique` is non-null. Shows:
- Supported conclusions as plain bullets
- Overreach concerns as warning cards with the specific quoted conclusion and the concern

The scorecard's four new rows render conditionally and use the same horizontal bar style as the existing rows.

The verdict card supports the new `RESULTS-PROBLEMATIC` value with its own color (recommend purple to distinguish from the existing five).

When the pipeline runs in v4 mode (no completed analysis), none of these sections render. The report looks exactly like a v4 report.

### 4.3 Flag UI — first-class implementation

This is the part that got lost in v4. v5 treats it as a primary feature.

**Three surfaces for flags:**

1. **Inline flag affordance on the Run page** — every classification displayed in the Evidence Table has a small flag icon. Clicking it opens a popover with: classification options, optional reason textarea, save button. No need to enter a separate review mode for one-off corrections.

2. **Per-run review mode** — accessible from the Run page via a "Review classifications" button. Opens a dedicated review surface:
   - One paper per card, full-width
   - Shows: title, abstract excerpt, year, venue, agent classification, justification quote, reasoning
   - Action row: keep / flag with correction dropdown / optional reason field
   - Keyboard shortcuts: `j` next, `k` previous, `f` flag, `c` change classification, `enter` confirm
   - Progress indicator at top: "12 of 47 reviewed, 3 flagged"
   - Auto-saves as the user moves through (no separate save action)

3. **Flag library page** — accessible from the main navigation:
   - List of all flags ever recorded
   - Filters: by domain, by classification correction type (e.g., "show only supports → contradicts corrections"), by date range
   - Each row shows: date, paper title (truncated), agent classification → user correction, reason
   - Clicking a row expands to show the full abstract excerpt and the original run it came from
   - Read-only — flags are immutable records
   - Search box for keyword search within abstract excerpts or reasons

**Live indicator during runs:**

When a run starts, if the flag store finds relevant past flags, the Run page shows a small badge near the progress timeline:

```
↻ 3 past corrections applied to this run
```

Hovering or clicking opens a tooltip listing the specific flags being injected. This is critical for trust — your father needs to see that his corrections actually affect the next run.

**Flag indicator on the home page:**

The home page sidebar shows total flag count and a link to the library. After enough flags accumulate, it can show "X flags from Y past runs."

### 4.4 History page — minor enhancements for v5

Already exists in v4. v5 adds:

- A small indicator on each history row showing whether that run was v4-mode (hypothesis only) or v5-mode (hypothesis + completed analysis). Different icons.
- The history detail view renders v5 critique sections when applicable.
- A "Re-run with new flags" button on each history row — re-runs the same input with the current flag store applied, so the user can see whether their accumulated corrections change the verdict.

### 4.5 What does not change in the frontend

- Vite + React + Tailwind + shadcn stack stays
- WebSocket event handling layer stays (only payload shapes change for two events)
- Progress panel design stays
- Tool call card design stays
- Charts and scorecard bar style stay
- Color scheme stays except for the new `RESULTS-PROBLEMATIC` verdict color
- No new routes — flag library is a sub-route under the existing navigation
- No new state management library

## 6. Things deliberately not in v5

Listing these explicitly so the scope stays disciplined.

**Not in v5:**
- gnomAD LOEUF integration — would help the reproducibility check verify more of your father's findings, but it's a meaningful new data source integration. Defer to v6.
- SynGO and BBB gene set auto-expansion — also v6. Worth doing but not what v5 is about.
- Scipy statistics step for the Analyst — also v6. The current Analyst still produces only Jaccard.
- Lineage-specific dN/dS (PAML/codeml integration) — defer indefinitely. The reproducibility check honestly reports that branch-specific values aren't verifiable rather than trying to compute them.
- Phylogenetic tree visualization — defer.
- Embedding-based flag retrieval — keyword overlap remains sufficient.
- Multi-user support, deployment, auth — out of scope permanently.
- Frontend tests — still low value at this stage.

**Worth noting for v6 planning:**

The single highest-value v6 addition would be integrating gnomAD + SynGO + a control gene set + scipy statistics. That moves the Analyst from "produces Jaccard" to "produces real Kruskal-Wallis on n=60+ genes per set with controls." That's the closest the tool can get to what your father did manually. Worth thinking about as the v6 north star but not part of v5.

## 7. Open questions to resolve before coding v5

Fewer than v4 since most architectural decisions carry over. Six questions:

1. **Severity grading for the Skeptic's critique.** The methods critique has severity (minor/moderate/major). Should the statistical and interpretation critiques also use severity badges, or are they binary (concern raised / not raised)? Recommend severity on all four — it gives the user a way to triage what matters most.

2. **Verdict precedence when both old and new categories apply.** If literature is `NOVEL-UNTESTED` but the user's completed analysis has serious methodological issues, what's the verdict? Recommend: `RESULTS-PROBLEMATIC` overrides `NOVEL-UNTESTED` when critique scores are low. Reasoning: the user has produced findings; if those findings have problems, that's the more actionable verdict.

3. **Can the user edit completed_analysis findings in the confirmation modal?** Editing a long structured findings block is fiddlier than editing a hypothesis textarea. Three options: (a) full structured editor with one card per finding, (b) raw text editor showing the JSON, (c) remove-only — user can drop the whole completed_analysis section or keep it as detected, no in-modal editing. Recommend (a) for usability, but it's the most build effort. Worth confirming before I commit.

4. **Reproducibility check scope.** Should the Analyst only check claims that match its existing retrieval capabilities (Ensembl dN/dS, ortholog counts), or should it also try to retrieve new data sources to verify claims (e.g., pull gnomAD on the fly for LOEUF verification)? Recommend: scope to existing capabilities in v5. Adding gnomAD on-the-fly is a v6 concern. Be honest in the report that some claims are "not currently verifiable by this tool."

5. **Flag review keyboard shortcuts — global or scoped?** When the user is in flag review mode, should `j`/`k`/`f` work globally or only when focus is on the review surface? Recommend: scoped. Users may scroll the surrounding UI and don't want every keystroke triggering navigation. Standard webapp keyboard handling.

6. **Flag library — local-only or with export?** Should there be an "Export flags as JSON" button for backup or sharing? Recommend yes — single line of code, useful for backup, and lays groundwork if you ever want to share flag libraries between users in a future product version.

## 8. What success looks like for v5

Concrete pass criteria:

**The combined pipeline works:**
1. Your father pastes a research proposal (no results yet). The tool detects this, treats as a pure hypothesis, runs v4 behavior, returns `NOVEL-UNTESTED`. Same as v4.
2. Your father pastes the same proposal plus his results document. The tool detects `completed_analysis`, shows the section-based confirmation modal, lets him verify each section.
3. After confirmation, the pipeline runs. The Skeptic activates critique mode.
4. The report renders both the v4 sections AND the four new critique panels.
5. The Skeptic catches at least these specific issues from his actual results:
   - Spearman r = -0.90 with n=4 modules has inadequate statistical power
   - No multiple testing correction across Kruskal-Wallis + Mann-Whitney + Spearman + Pearson
   - Phylogenetic non-independence not addressed in the comparative dN/dS
   - The "two waves" finding doesn't account for divergence-time uncertainty
6. The reproducibility check honestly reports which of his statistics can be verified against Ensembl and which can't.
7. The verdict resolves to either `MODERATE` (hypothesis defensible, results have issues) or `RESULTS-PROBLEMATIC` (results too compromised to draw conclusions).

**The flag UI works:**
1. Your father reviews the classifications inline on the Run page. He clicks a flag icon, picks the correct classification, optionally adds a reason. The flag saves.
2. He opens the flag library and sees his recent flag listed.
3. He re-runs the same hypothesis. The Run page shows "1 past correction applied to this run."
4. He can verify in the library that the flag was used.

If 1–7 work in the pipeline test and 1–4 work in the flag test, v5 succeeds.

## 9. Risk and what I'm less sure about

Three things worth flagging honestly:

**One — the completed_analysis extraction may be unreliable.** Real research documents are messy. The Formalizer will sometimes misclassify a "preliminary observation" as a completed finding, or miss subtle results buried in a methods section. The confirmation modal mitigates this but doesn't eliminate it. The user has to actually look at the modal and catch mistakes. If they rush through and click confirm, errors propagate.

**Two — the critique block's quality depends entirely on the Skeptic agent's domain knowledge.** Claude Sonnet 4 is good at general statistical reasoning, but specific biology methodological knowledge (PGLS, phylogenetically-corrected dN/dS, the right way to date gene emergence) is at the edge of its training. I expect the critique to catch the obvious issues (small n, missing multiple testing) reliably and to miss subtler ones (e.g., specific issues with branch-models in PAML). That's still useful, but it's not a substitute for actual biostatistical review.

**Three — reproducibility check has narrow scope.** v5's reproducibility check only verifies Ensembl-retrievable data. Most of what your father reported (LOEUF, primate-rodent acceleration, two emergence waves) isn't verifiable through this tool. The check will honestly report this, but it means a lot of his analysis stays in the "we can't verify this" bucket. The honest framing in the report matters — we have to avoid implying that "not verifiable" means "wrong."

## 10. What v5 isn't trying to be

This is worth being clear about. v5 doesn't make the Nullifier a replacement for a proper biostatistical methods reviewer. It doesn't make it a peer reviewer. It doesn't make it a substitute for running the analyses in PAML or HyPhy yourself.

What v5 does is close the workflow loop: a single tool that can stress-test a claim at any stage of investigation, learn from corrections, and surface the most common methodological issues that researchers cut corners on. That's a useful tool. It's not the whole pipeline of scientific review, but it's the part that AI can plausibly do well.

If v5 ships and your father uses it on a real analysis and it catches one significant issue he hadn't thought about, that's a win. If it catches several, that's a strong product signal worth investing further in.
